# What is Dovetail?

![Dovetail overview diagram](https://dummyimage.com/1200x560/0b1220/f8fafc.png&text=Dovetail+Overview+Architecture)

The goal of Dovetail is to be a simple way to bridge the gap between Configuration driven development and Dovetail driven development.
In an ASP.NET Core application it is "pay-for-play" in that you must register all your configurations, services, etc. This is great but it
also forces you into cases where if you forget to call an extension method nothing lights up like you expected. Another draw back of using
extension methods is that any dependencies you need, to configure services for example, must be included in the method call.

One thing to keep in mind is that ASP.NET Core just got rid of the [double container](https://github.com/aspnet/Announcements/issues/353)
problem, we are not introducing a second container, instead we just implement `IServiceProvider` using the same [dictionary that is provided
by `IHostBuilder.Properties`](https://docs.microsoft.com/en-us/dotnet/api/microsoft.extensions.hosting.ihostbuilder.properties). Any services
that you inject into your conventions must exist inside of this dictionary, or be optional.

At a high level Dovetail solves this problem in a very similar way to something like [Autofac Modules](https://autofaccn.readthedocs.io/en/latest/configuration/modules.html)
or other similar assembly scanning solutions. Our goal is to be agnostic at a high level, strictly speaking using MSDI/MSEL/MSC is not required however is currently
assumed that .NET Platform Extensions are going to be used. In the future this might be broken out.

## Why use Dovetail?

You have to make a choice with every library you introduce, and using Dovetail is no different. If it's not for you, that's okay! Here are some possible reasons why and why not to use Dovetail.

### Reasons to use?

- You can create library code that "just works" without having to ensure that the correct extension method was called, with the correct parameters, for every library you add.
- You can don't have to worry about service registration extension methods  in instead each module can setup it's own registrations.
- You can can group bootstrapping logic together to lessen cognitive load.

### Reasons not to use?

Everything has it's positives and negatives. We cannot lie using conventions will add time to starting up your application. There is a trade off to be made here
and optimizations that can be made to lessen that trade-off.

Fundamentally if you're making a highly focused, and specialized microservice then using Dovetail may not be for you. You will own the burden
to ensuring that configuration is done the same, extension methods are called, and everything else. It is called "pay-to-play" after all.

#### Note on performance

A quick note on performance. We are now using Source Generators to generate export methods, that return all conventions for a given assembly. This allows convention
loading to have similar performance characteristics to pure extension methods, it's still slower but reduces the need to assembly scan.

The <xref:Dovetail.ICompiledTypeProvider> is now generated by source generator at compile time, further reducing the need to assembly scan at runtime.

# What is a Dovetail

A convention is nothing but a class that implements the <xref:Dovetail.Infrastructure.IDovetailJoint> interface. Nothing special.
By itself <xref:Dovetail.Infrastructure.IDovetailJoint> is just a marker interface used to identify a class that may implement one more conventions.

Dovetails are then interfaces that are implemented by the class some examples are:

- <xref:Dovetail.Configuration.IConfigurationJoint>
- <xref:Dovetail.DependencyInjection.IServiceJoint>
- <xref:Dovetail.Logging.ILoggingJoint>
- <xref:Dovetail.Setup.ISetupJoint>
- <xref:Dovetail.Hosting.IHostingDovetail>

ad-hoc Dovetails (or delegate Dovetails) are a different kind of convention but written as a delegate, but they mirror the method given by the Dovetail interface.
These can be used to during program startup just like `ConfigureServices` are used today.

Some examples are:

- <xref:Dovetail.Configuration.ConfigurationJoint>
- <xref:Dovetail.DependencyInjection.ServiceJoint>
- <xref:Dovetail.Logging.LoggingJoint>
- <xref:Dovetail.Setup.SetupJoint>
- <xref:Dovetail.Hosting.HostingDovetail>

## Exporting a Dovetail

A convention can be exported by using the <xref:Dovetail.DovetailAttribute> (eg. `[assembly: Dovetail(typeof(MyDovetail))]`).
This is important you may want to make conventions are **not** enabled by default but are available to be loaded by users if they want to.

The rules are straight forward.

- A public convention - Will not be loaded unless it is added to the <xref:Dovetail.DovetailContextBuilder>.
- A public exported convention - Will always be loaded unless excluded by assembly or type in the <xref:Dovetail.DovetailContextBuilder>.
- A private convention - Will never be loaded. ever.
- A private exported convention - Will always be loaded unless excluded by assembly in the <xref:Dovetail.DovetailContextBuilder>.
